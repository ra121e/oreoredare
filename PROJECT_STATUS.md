## 8. 今後の計画

今後は、見た目の拡張よりも、**MVP の中核である「実際に怪しい電話を扱える流れ」を成立させること**を優先します。

### 実装方針の更新

実機スマホでホーム画面と着信オーバーレイのモック表示、および基本的なボタン操作の確認が完了したため、今後は UI の細かな見た目調整をいったん後回しにします。

この最小パスは成立したため、次のフェーズでは **「制約の整理と適用範囲の拡大」** を優先します。

今回の方針で後回しにする項目は以下です。

- 色味や余白の細かな調整
- アニメーションの磨き込み
- 文言の最終調整
- 履歴画面や家族画面の見た目改善
- AI 自動応答、声紋判定、SMS 解析、Supabase 連携の本格接続

次に到達すべき目標は、**「AQUOS sense8 で、未登録番号だけでなく必要に応じて連絡先登録済み番号や前面外の状態でも、どこまで安定して着信ガードを成立させられるかを整理すること」**です。

### 直近の優先順位

1. 連絡先登録済み番号への対応方針を決め、必要なら `READ_CONTACTS` を追加する
2. バックグラウンドやロック画面での制約を確認し、必要なら次の方式を選ぶ
3. Call Screening role の付与手順を簡潔な実機手順として整理する
4. その後に高齢者向け画面、家族連携、判定機能を段階的に広げる
5. UI の最終調整は実動フロー確認後に行う

### フェーズA: 開発基盤の整理

目的: 実装と検証を止めない状態を維持する

- `.dart_tool` 配下の権限問題を再発させない運用にする
- `README.md` と `pubspec.yaml` の初期文言をプロジェクト内容に合わせて更新する
- `PRODUCT.md` と `PRD.md` の重複を整理する
- ドキュメントと実装の対応関係を簡単に追えるようにする

### フェーズB: 高齢者側 MVP の完成度を上げる

目的: 高齢者が実際に触る主要体験を UI と状態遷移で固める

- ホーム画面を実データ前提の構造に整理する
- 履歴画面を一覧表示付きの実用形にする
- 家族連絡画面をワンタップ連絡前提で作り直す
- 着信オーバーレイを実イベント連携しやすい構造に調整する
- 音声ガイドの発火タイミングと文言を整理する

備考:
このフェーズの見た目改善は、実着信との接続が成立したあとに優先度を上げる。

### フェーズC: Android 通話連携の着手

目的: このアプリの価値の中心である「着信時ガード」を動かし始める

- Call Screening role を端末で付与し、AQUOS sense8 で未登録番号に対する実動作確認を完了した
- 連絡先登録済み番号を対象に含めるかを決め、必要なら `READ_CONTACTS` と許諾フローを追加する
- バックグラウンドやロック画面での挙動を確認する
- 必要なら `InCallService` など次の Android ネイティブ方式を検討する
- Flutter 側のイベント処理を実機結果に合わせて微調整する
- 最初の到達点だった「着信があったら AI 応答中オーバーレイを表示する」は達成済み
- 次は「どの着信条件で安定して通るか」を明確にする

備考:
このフェーズでは、AI 自動応答、通話ブロック、声紋判定、SMS 解析はまだ実装対象に含めない。
初回実装では `CallScreeningService` を incoming_call の主経路として採用し、
`PhoneStateListener` は補助経路として組み合わせた。
今後、AQUOS sense8 での実機結果しだいで `InCallService` まで広げるか判断する。

### フェーズD: 家族連携とバックエンド基盤

### フェーズE: 判定機能と安全機能の実装

目的: 「見た目のアプリ」から「守れるアプリ」へ移行する

- 合言葉照合
- 声紋登録と照合
- 詐欺キーワード判定
- ブロック時ログ保存
- 家族への通知
- 通話録音と要約の方針決定
